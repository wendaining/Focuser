# Focuser i18n Guide

This document explains how Focuser's UI translations are wired and how to add or update translation keys safely.

## Files

The UI uses a small custom i18n layer:

- `crates/focuser-ui/ui/i18n.js` loads translations, resolves keys, interpolates parameters, and applies text to the DOM.
- `crates/focuser-ui/ui/i18n/en.json` is the English source translation file.
- `crates/focuser-ui/ui/i18n/zh.json` is the Simplified Chinese translation file.

Every supported locale must have a JSON file named `<locale>.json` in `crates/focuser-ui/ui/i18n/`. The locale code must also be accepted by `i18n.init()` and the language buttons in `index.html`.

## Key Structure

Translation keys are dot-separated paths into the JSON object. For example:

```json
{
  "settings": {
    "language": "Language",
    "languageDesc": "Interface display language"
  }
}
```

These values are read as:

```js
t('settings.language')
t('settings.languageDesc')
```

The JSON hierarchy is only for organization. The runtime walks each dot-separated segment, so `settings.language` means:

1. Open the top-level `settings` object.
2. Read its `language` string.

Keep related strings grouped under the same top-level object. Current common groups include:

- `nav`: sidebar and navigation labels.
- `common`: generic reusable UI words and short phrases.
- `toast`: toast messages.
- `confirm`: confirmation dialog titles and messages.
- `modal`: modal labels and copy.
- `dashboard`, `blocklists`, `websites`, `apps`, `schedule`, `statistics`, `settings`: page-specific strings.
- `status`, `pomodoro`, `allowance`, `updater`: feature-specific strings.

## JSON File Rules

All locale files must have the same key shape. If `en.json` has `settings.language`, then `zh.json` must also have `settings.language`.

Use strings as leaf values:

```json
{
  "common": {
    "save": "Save"
  }
}
```

Do not use arrays, numbers, booleans, or nested objects as final translated values. `i18n.t()` only returns values that are strings.

Keep JSON valid:

- Use double quotes for keys and strings.
- Do not include comments.
- Do not leave trailing commas.
- Escape literal double quotes inside strings as `\"`.
- Preserve placeholders exactly, including their names and braces.

## HTML Fields

Static DOM text is connected to JSON keys through `data-i18n` attributes.

```html
<span data-i18n="settings.language">Language</span>
```

This means:

- The fallback text in HTML is `Language`.
- On load or language switch, `i18n._applyToDOM()` reads `settings.language`.
- The element's `textContent` is replaced with the translated string from the active locale JSON file.

Use the attribute that matches the field being translated:

```html
<span data-i18n="common.save">Save</span>
<input data-i18n-placeholder="websites.searchWebsites" placeholder="Search websites...">
<button data-i18n-title="common.delete" title="Delete"></button>
<p data-i18n-html="schedule.blocking247Desc">
  This block list is enforced at all times.
</p>
```

The mapping is:

- `data-i18n`: sets `textContent`.
- `data-i18n-placeholder`: sets `placeholder`.
- `data-i18n-title`: sets `title`.
- `data-i18n-html`: sets `innerHTML`.

Use `data-i18n-html` only when the translation intentionally contains simple markup such as `<strong>`. Do not use it for user-provided data.

## JavaScript Fields

Dynamic UI built in `app.js` should use `t(key)` directly:

```js
container.innerHTML = '<div class="empty-state">' + t('dashboard.noActivity') + '</div>';
```

If the string contains user-provided content, escape that content before inserting it into HTML:

```js
container.innerHTML =
  '<div>' + t('common.noMatch', { query: escapeHtml(query) }) + '</div>';
```

For plain text assignment, use `textContent`:

```js
label.textContent = t('schedule.selectBlockList');
```

## Parameters

Translations support named placeholders using double braces:

```json
{
  "toast": {
    "imported": "Imported {{count}} {{item}}"
  }
}
```

Call it with a parameter object:

```js
t('toast.imported', { count: result.added, item: 'domains' })
```

Every locale must keep the same placeholder names. Translators may move placeholders to fit the target language, but they must not rename or remove them unless the JavaScript call is also updated.

Good:

```json
{
  "toast": {
    "imported": "Imported {{count}} {{item}}"
  }
}
```

```json
{
  "toast": {
    "imported": "已导入 {{count}} 个{{item}}"
  }
}
```

Bad:

```json
{
  "toast": {
    "imported": "已导入 {{total}} 个{{name}}"
  }
}
```

The bad version will not receive values because the JavaScript passes `count` and `item`, not `total` and `name`.

## Adding a New String

When adding a new user-facing string:

1. Choose a key under the most specific existing group.
2. Add the English string to `en.json`.
3. Add the matching key to every other locale file.
4. Reference the key from HTML or JavaScript.
5. Keep the visible fallback text in HTML readable, usually English.

Example for static HTML:

```json
{
  "settings": {
    "syncNow": "Sync now"
  }
}
```

```html
<button data-i18n="settings.syncNow">Sync now</button>
```

Example for dynamic JavaScript:

```json
{
  "toast": {
    "syncFailed": "Sync failed: {{error}}"
  }
}
```

```js
toast(t('toast.syncFailed', { error: String(e) }), 'error');
```

## Naming Guidelines

Use stable semantic names, not names based on the current English wording.

Good:

```json
{
  "settings": {
    "deleteAllData": "Delete all data"
  }
}
```

Avoid:

```json
{
  "settings": {
    "deleteAllButtonRedText": "Delete all data"
  }
}
```

Prefer:

- `title`, `subtitle`, `desc`, `heading`, `empty`, `placeholder`, and `hint` for common page copy roles.
- Specific names for actions, such as `exportConfig`, `clearStatistics`, or `startFocusSession`.
- Existing group names before creating new top-level groups.

Avoid duplicating strings unless they may need different translations in context. For generic labels like "Save", "Cancel", or "Delete", use `common`.

## Fallback Behavior

If a key is missing or its value is not a string, `i18n.t()` returns the last segment of the key. For example, `settings.missingKey` displays `missingKey`.

This fallback is only a debugging convenience. Missing keys should be fixed before release.

## Locale Defaults

New installs default to English when no `ui_language` setting exists. The selected language is stored through the Rust backend setting key:

```text
ui_language
```

Changing this setting changes which JSON file is loaded on the next initialization.

## Review Checklist

Before committing i18n changes, check:

- Every new key exists in all locale JSON files.
- Every placeholder name is identical across locales.
- HTML uses the correct attribute for the target field.
- Dynamic JavaScript uses `t()` for user-facing text.
- User-provided content is escaped before being inserted through `innerHTML`.
- JSON files parse successfully.
